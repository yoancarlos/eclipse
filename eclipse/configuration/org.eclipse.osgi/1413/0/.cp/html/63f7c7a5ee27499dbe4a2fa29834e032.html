Testing the Integration Flow Configuration

Prerequisites

  • You have configured the integration flow with values required for implementing a scenario.

Context

You perform this task to test if an input message, from the sender, gets processed through the configured elements of the integration flow, such as message mapping, operation mapping content based router, splitter, etc., and results to an expected output message at the receiver. You can cover testing for any type of integration flow elements by using the advanced testing techniques such as passing additional test parameters and bundles or mocking request or response messages for synchronous calls.
Restriction Test view does not support the testing of the following:
  • Secuity elements such as Signer, Verifier, Encryptor or Decryptor, but the testing features allow you to mock the security elements with Encoder/Decoder to test the data flow in its entirety. For example, Signer or Encryptor is mocked with Encoder, and Verifier or Decryptor is mocked with Decryptor
  • XML Digital Signature or XML Signature Verifier
  • Enricher element that has an expression containing custom beans.
Note If you are facing any issues while testing the integration flow configurations, then find the troubleshooting information in the wiki page.

When the test execution of the integration project is complete, a new test package with the following structure gets added to the integration project in the Project Explorer view:

Table 1: Description of the Testing Package
Content of Testing Package Description
test.configuration contains test case in XML format
test.content contains autogenerated java code or allows you to create JUnit test cases under it
test.data contains input and output files

Procedure

  1. Open the integration flow in the Model Configuration editor.
  2. In the main menu, select Window > Show View > Other....
  3. In the Show View dialog, select Other… > Test View.
  4. Provide an input payload XML for the integration flow configurations that you are testing.
    Note You can provide the input payload XML in one of the following ways:
    • From the toolbar of the Test View, choose (Generate Input icon) to generate input XML for the integration flow synchronized with the Test View.
      Tip You can generate input payload XML only if you have assigned sender interface and saved the integration flow. For more information on how to assign interface, see Assigning Interfaces.
      Restriction The input payload XML cannot be generated for:
      • A configuration having multiple senders
      • A WSDL having multiple operations defined
    • If you want to use an existing input payload XML, choose the dropdown arrow next to the (Generate Input icon) from the toolbar of the Test View and select (Browse Input Payload icon) to browse for an <input>.xml file.
    • Manually type the input payload XML in the Input XML section.
    • Drag an existing XML file from a test.data package and drop it in the Input XML section.
  5. If you want to test the configurations with more inputs for an overall testing of a scenario choose (Set Testdata) .
    Note You can test a scenario by injecting additional inputs, such as, passing additional headers and configurations, or value mapping bundles.
    • In case you want to test if headers required by your scenario are retained, you can pass the additional headers that you have already configured in the Allowed Headers field of the integration flow.
    • In case of testing an operation mapping, you can define a mock response or a fault response.
    • In case of value mapping, you can pass the value mapping project details or value mapping bundle.jar
  6. If you want to update the runtime target platforms and common testing libraries that are shipped by SAP, choose the dropdown arrow next to (Set Testdata) from the toolbar of the Test View and select (Reload Runtime Libraries).
    Note
    • The Reload Runtime Libraries option updates only the default runtime target plaftorm offered by SAP and not the additional runtime target platform that you specify in the Runtime Platform page in Testing Integrations under Preferences.
    • If you want to set additional runtime target platforms apart from the default runtime target platforms, see Setting the Target Platform for Testing.
  7. If you want to set the preferences to be used during testing, choose the dropdown arrow next to the (Set Testdata) from the toolbar of the Test View and select (Set Testing Configurations) from the toolbar of the Test View.
  8. If you want to select a particular runtime target platform on which the test should be executed, then perform the following substeps:
    1. Choose the dropdown arrow next to (Set Testdata) from the toolbar of the Test View and select Set Target Platform option.
    2. In the Set Target Platform dialog, select the required target platform from the Runtime dropdown.
      Note The runtime target platforms available in the dropdown list correspond to the runtime target platforms you have entered in the Runtime Platform page in Preferences. If you want to add more runtime target platform, choose Add....
  9. Choose (Execute Test) from the toolbar of the Test View to run the test and obtain the output at the Output XML section.
    Note
    • If you have not specified any runtime target platform or testing libraries in the Preferences page or if the specified update sites do not have any content, then the runtime target platform and common testing libraries are automatically downloaded from a default update site during test execution.
    • If you want to check the header information from a response message, then deselect the Do not show header in ouput payload option in the Preferences page.
    • If you are testing a content-based router with only a default receiver, then no output xml can be generated.
    • If the test execution of the integration flow scenario is not resulting to any <output>.xml, you can retry execution after increasing the value for Duration for executing test(sec) option in the Preferences page.
    • If you want to debug the test execution steps, deselect the Disable runtime log trace during test execution to view detail runtime logs in the Console.
  10. In the Test View, under the Output XML section, select a receiver from the dropdown to check the output at that receiver. This is useful for integration flows with multiple receivers.
  11. If you want to save the configurations as a test method to a test case, then follow the steps below:
    1. From the toolbar of the Test View, select (Save as Test Configuration).
    2. In the Save Configurations as Test Case wizard, enter a name for the new test case or select an existing one.
      Restriction The test method saves additional testing configurations such as value mapping bundles, headers, mock response and fault messages that are passed to the integration flow during testing but such parameters are not considered during JUnit test case execution.
      Note You can view this test case as <test case name>.ifltest under the test.configuration package in the Project Explorer view. The input and output xmls are available in the test.data package under testing folder.
    3. Enter a name for the new test method or select an existing test method.
    4. Choose Finish.
      Note If you add this configuration as a new test method, then you can see this gets included in the <test case name>.ifltest as a method. If you select an existing test method, then the test method content gets overwritten with the new configuration.
  12. If you want to load an existing configuration, select the dropdown arrow next to (Generate Input icon) and choose (Load Test Configuration icon) from the toolbar of the Test View.
    Tip

    The Load Test Configuration option is useful when you want to retest the integration flow with previously saved configurations that include input XML and additional parameters.
